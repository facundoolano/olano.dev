As I worked on my [[style-guide][programming style guide]], channeling my inner code reader, recalling what is it like to dive into an unfamiliar codebase---to look for clues in its components and effectively extract knowledge from them---a particular formulation came to mind: /what is this/ and /why is it here?/. I realized that looking for answers to those questions, and supplying them myself, makes a large part of my work on software projects. These questions apply to artifacts of all sizes, from humble functions to entire projects, from a pull request description to the user story that it (hopefully) originated from. Behold yet another hyperbolic definition of the software development process: to answer /what is this/ and /why is it here/, over and over.

The what and why formula is just a specific instance of one of Ousterhout's principles: /code should be obvious/. The harder to find answers---the further the answers are from the piece of code in question---the likelier that something's missing or something needs to be removed. Maybe that module needs a top-level docstring to explain its purpose, or a link to relevant documentation; maybe this new function could be replaced or absorbed by a preexisting one; maybe engineers didn't ask enough questions to the product folks to clarify the intent of this user story; maybe we don't need this feature at all.

When an artifact doesn't readily provide the /what/ and the /why/, the engineer needs to make an extra effort to go somewhere else looking for answers[fn:1]. When those inquiries yield results, it's the engineer's job to incorporate that newly found knowledge into the project, perhaps reorganizing the code while at it, to (again using in Ousterhout's parlance) /emphasize what matters/, making the questions easier to answer next time around.

One way to emphasize what matters is to make it prominent, and the most prominent things are those that come first, so I've been making efforts, when I write code, to make it readable top-to-bottom, placing important things---those that provide the /what/ and the /why/---at the top, pushing details below, assuming that most readers shouldn't need to go past the first few lines of a module to find what they are looking for.

Imagine a language designed to encourage programmers, perhaps even force them, to constantly think about and provide answers to those questions[fn:2].

The core feature of this language would be the ~why:~ statement, a form of docstring required on all modules and module exports. If this sounds like an annoyance, it would be, but a useful one: hard to think about a reason to introduce this new module? Maybe there isn't a good one. Hard to explain why this function needs to be exported? Then maybe it shouldn't. Suddenly there's a force pushing you to build deeper modules.

Much like in Erlang, modules would declare what they export, and these declarations would go at the top of the file, before imports, because exports make the interface, and imports are implementation details. Incidentally, each import statement should also be accompanied by its ~why:~. Are you sure you need to bring in that external piece of code? Are you sure it really needs to be external?

We could even have some of those pesky LLM review bots check that the ~why:~ docs are actually giving plausible answers and not just lazily reiterating the code.

What would that look like?
#+begin_src rust
export struct Game {
    why: "Holds the entire game state, including the hero character and its current location.
          Its methods implement rpg-cli's core gameplay: moving across directories and engaging in battles."

    fn new() -> Self
        why: "starts a game, rolling a new character."

    fn reset(&mut self)
        why: "resets the current game, rolling a new character."

    fn go_to(mut self, dest: &Location)-> Result<(), character::Dead>
        why: "core gameplay of the game: move the hero's location towards the given destination,
              one directory at a time, with some chance of enemies appearing on each one."

    player: Character      why: "tracks hero status and equipment."
    location: Location     why: "tracks current hero location in the filesystem."
    gold: i32              why: "tracks accumulated gold, usable to buy items."
    inventory: Vec<Item>   why: "tracks collected and currently unequipped items."
}

impl Game {
    import characters      why: "To define the hero and spawned enemies."
    import location        why: "To track and update the hero location."
    import items           why: "inventory, battle rewards, hero equipment."
    import randomizer      why: "To randomly spawn enemies and yield battle rewards."
    import json            why: "for game data serialization/deserialization."

    fn new() -> Self
    {
        Self {
            player: Character::player(),

            inventory: HashMap::new()
        }
    }

Problems:
- When a module purpose is to hold a single struct, as in the example above, it's probably redundant to justify the module and the struct.
- Some code is self explanatory, do we really need to always explicitly justify it? Isn't that like those old java shops that forced a docstring on getters and setters? Maybe we need an ~obvious~ keyword to bypass documentation.
- Reading the code to understand the system is a different activity from scanning the code in order to update it. Maybe I just want to see a straight struct definition and not its documentation.
- Public function declaration duplicates need to be reiterated in the implementation, which smells like C++ header files. That two-way syncing should be done by a machine.
- Similarly, during development it's more convenient to inline the docstrings in the function definitions rather than their top-level declarations.

Should we have separate reader/writer modes of modules? Reader mode being similar to what the Python shell shows upon ~help()~ invocations? Do we need smarter fmt tools to more heavily rearrange the code, optimizing it for readability? As machine generated code prevails, should we move towards interface-focused reviews and pay less attention to implementation? Is this [[http://www.literateprogramming.com/knuthweb.pdf][literate programming]]?
